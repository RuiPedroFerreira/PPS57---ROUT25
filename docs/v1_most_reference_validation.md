# V1 — TSP method port onto the MoST reference network

V1 of the validation plan (the V0 harness, added in a separate PR) answers the *method* question
("does the TSP engine decide correctly?") independently of the *model* question
("does our Porto demand imitate reality?"). It runs the existing TSP engine +
Safety Layer on a **real, peer-reviewed, externally-built reference network**
instead of the synthetic Porto corridor — so a failure here is the method's,
not our calibration's.

## Source of truth

[MoST — Monaco SUMO Traffic](https://github.com/lcodeca/MoSTScenario), a published
multimodal C-ITS reference scenario. It is **downloaded, not vendored** (MoST is
GPL-3.0): `scripts/fetch_most_scenario.py` pulls only the network file at a pinned
commit and verifies its hash.

produced from exactly the pinned artifact.

## Reproduce

```bash
.venv/bin/python scripts/fetch_most_scenario.py        # download + verify pinned MoST net
.venv/bin/python scripts/run_most_reference_check.py   # probe every TLS, write the report
```

The runner drives the pre-existing `scripts/empirical_network_profile_check.py`
(one isolated TraCI run per traffic light) and consolidates the results. It
invents nothing — every row is a real run against the pinned network.

## Findings (`docs/validation/v1_most_method_check.json`)

- **12 / 12 traffic lights** in the real MoST network were profiled and probed
  end-to-end through the TSP engine and Safety Layer.
- **0 profile mismatches** on every one: the map-agnostic `NetworkProfile`
  reproduces SUMO's loaded TLS programs (phase states, durations, controlled-link
  counts) exactly, on a network generated by an unrelated `netconvert` run in
  2019. This is the core V1 result — the engine's view of the world matches what
  SUMO actually loaded.
- The engine produced real priority decisions (e.g. `early_green` —
  *truncate_conflicting_phase_to_anticipate_priority_movement_green*) on real
  discovered movements with real conflict sets.
- The **Safety Layer engaged on every intersection** and blocked. In the committed
  run, at the fixed 30 s probe time, all 12 report `min_green_not_satisfied:4.0<8.0`
  (the probe arrives mid-min-green). Across other probe times the reason varies with
  phase timing (e.g. `current_phase_is_yellow`,
  `green_extension_requires_priority_movement_green_phase`). **A block is the correct
  outcome** when the probe's arrival coincides with unsafe timing — it is not a failure.

**Verdict: pass** — the criterion is zero profile mismatches and end-to-end
execution on a real reference network, both met.

## Honest scope and limits

- V1 validates **method generalization and profile fidelity**, not demand or
  outcome realism. The probes use a synthesised priority request and the generic
  auto-discovered safety constraints (`min_green 8 s`, etc.), so they characterise
  *that the pipeline runs correctly on real geometry*, not how much delay TSP
  saves. That is V5's job, against STCP AVL and the TSP face-validity envelope.
- Under fixed probe timing + conservative generic constraints, the Safety Layer
  blocked all 12 probes. The approved → actuated path is exercised separately by
  the Porto demonstrator and the actuation tests; reproducing an approval on MoST
  would require timing each probe into a valid green window, which we deliberately
  do **not** tune for (it would be cherry-picking).
- MoST TLS are static-program here; real adaptive/actuated control and real
  signal plans are V4.
